## Suggestion 1 — Extract Duplicated Symlink Management into a Shared Utility

### Why

`deployer.rs` and `rollback.rs` each contain an identical pair of code blocks for
removing and creating the `current` symlink.  The platform-branching logic
(`#[cfg(unix)]` / `#[cfg(windows)]`) is copy-pasted verbatim across both files
(deployer.rs lines 70–95 and rollback.rs lines 82–106, and again at lines
126–152).  Any future bug fix or new platform support must be applied in every
copy, which is error-prone.

### How

1. Create `src/utils.rs` (or `src/fs_utils.rs`) and expose two public helpers:
   ```rust
   pub fn remove_symlink(path: &str) -> Result<(), Box<dyn std::error::Error>>;
   pub fn create_symlink(target: &str, link: &str) -> Result<(), Box<dyn std::error::Error>>;
   ```
2. Replace the duplicated `#[cfg(unix)]` / `#[cfg(windows)]` blocks in both
   `deployer.rs` and `rollback.rs` with calls to these helpers.
3. Add the new module to `lib.rs` and `main.rs`.

**Scope of change:** `src/deployer.rs`, `src/rollback.rs`, new `src/utils.rs`,
`src/lib.rs`, `src/main.rs` (module declaration only).

### Risk

**Low.** This is a pure structural extraction with no behavioral change.
The only risk is an unintentional logic difference during the copy if done
manually — mitigated by covering the helpers with unit tests before swapping
call sites.

## Suggestion 2 — Extract Duplicated Command Execution into a Shared Helper

### Why

`builder.rs` (`build` function, lines 8–32) and `deployer.rs`
(`deploy_with_command` function, lines 9–34) share an identical pattern:

1. Split a `&str` command on whitespace.
2. Take `parts[0]` as the program and `&parts[1..]` as arguments.
3. Run `Command::new(program).args(args).current_dir(repo_path).output()`.
4. Check `output.status.success()`, log stderr on failure, log stdout on
   success.

Keeping this duplicated means a future change (e.g., shell quoting support,
timeout handling, or structured error types) must be applied twice and risks
divergence.

### How

1. Add a `run_command` function to the utility module from Suggestion 1 (or to a
   dedicated `src/process.rs`):
   ```rust
   pub fn run_command(
       command: &str,
       cwd: &str,
   ) -> Result<(), Box<dyn std::error::Error>>;
   ```
2. Replace the body of `builder::build` and `deployer::deploy_with_command` with
   a single call to `run_command`.

**Scope of change:** `src/builder.rs`, `src/deployer.rs`, shared utility module.

### Risk

**Low.** The two existing code paths are functionally identical, so extraction
does not alter behavior.  Risk is limited to compile-time breakage if the
signature is not matched exactly — caught immediately by `cargo build`.

## Suggestion 3 — Make the Config File Path a Top-Level CLI Argument

### Why

The string `"deploy.toml"` is hard-coded in five separate functions in
`main.rs` (`cmd_init`, `cmd_run`, `cmd_rollback`, `cmd_status`, `cmd_log`).
Users who want to maintain multiple configurations (e.g., staging vs.
production) or who store the file outside the working directory have no way to
specify an alternate path without recompiling.  The repetition also means the
string must be updated in five places if the default name ever changes.

### How

1. Add a global `--config` / `-c` option to the `Cli` struct:
   ```rust
   #[derive(Parser)]
   #[command(name = "postloop")]
   struct Cli {
       /// Path to the configuration file (default: deploy.toml)
       #[arg(short, long, default_value = "deploy.toml")]
       config: String,

       #[command(subcommand)]
       command: Commands,
   }
   ```
2. Pass `cli.config` (or a reference to it) into each `cmd_*` function,
   replacing the five hard-coded string literals.

**Scope of change:** `src/main.rs` only.

### Risk

**Medium.** Adding a new flag is backward-compatible because the default value
preserves existing behavior.  The risk is that integration tests or scripts that
inspect the exact `--help` output may need updating.  Any test that currently
calls `postloop run` without `-c` will continue to work unchanged.
